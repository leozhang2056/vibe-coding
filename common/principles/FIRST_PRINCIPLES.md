# 第一性原则

> First Principles Thinking — 从基本事实出发推理，而非从类比或惯例出发。

## 核心思想

第一性思维要求：

1. **拆解到基本事实**：把问题分解到不可再分的、可验证的真理
2. **质疑假设**：不默认「别人都这么做」就是对的
3. **从零推导**：在基本事实之上重新构建方案，而不是照搬既有实现

在编程与架构中，用于避免盲目跟风、过度设计，并让方案与真实需求对齐。

---

## 原则

### 1. 先问「问题本质是什么」

- 不急于选框架、选模式，先明确要解决的核心问题
- 区分「必须满足的约束」与「习惯或惯例」

### 2. 从约束与目标反推设计

- 性能、一致性、可维护性等，哪些是硬约束？
- 在满足约束的前提下，选择最简单、可验证的实现路径

### 3. 质疑「我们一直这样写」

- 现有代码和流程是否仍然符合当前目标？
- 若仅因历史原因存在，可考虑简化或删除

### 4. 小步验证再扩展

- 用最小可行方案验证核心假设
- 在验证通过后再增加抽象与复用

## 在编程中的运用

### 需求阶段

```
❌ 类比思维：「别的系统都有消息队列，我们也上 Kafka」
✅ 第一性：当前 QPS、延迟要求、团队运维能力是什么？是否真的需要分布式消息队列，还是内存队列/数据库即可？
```

### 架构与选型

```
❌ 类比思维：「微服务是趋势，我们就拆成 10 个服务」
✅ 第一性：领域边界、团队规模、部署与排障成本如何？是否先有清晰模块边界再谈进程/服务拆分？
```

### 代码实现

```
❌ 类比思维：「这里大家都用设计模式，所以上个工厂 + 策略」
✅ 第一性：这里变动的轴心是什么？若只有 2～3 种稳定分支，简单 if/策略即可，不必为「模式」而模式。
```

## 示例

### 示例 1：缓存要不要上？

```python
# 第一性提问：
# - 读 QPS、数据变更频率、可接受延迟？
# - 若 QPS 低、数据常变，多一层缓存反而增加一致性问题

# ❌ 盲目跟风：别人都用 Redis，我们也上
def get_user(user_id):
    cached = redis.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)
    user = db.query(User).get(user_id)
    redis.setex(f"user:{user_id}", 3600, json.dumps(user))
    return user

# ✅ 第一性：先验证是否真的需要
# 若只是单机、低 QPS，内存 dict 或 lru_cache 足够
from functools import lru_cache

@lru_cache(maxsize=1024)
def get_user(user_id: int):
    return db.query(User).get(user_id)
```

### 示例 2：接口设计

```python
# 第一性提问：
# - 调用方真正需要哪些字段？变更频率如何？
# - 是否必须一次返回「整个聚合」，还是按场景拆分接口更稳？

# ❌ 惯例思维：用户接口就返回「用户全部信息」
def get_user_profile(user_id):
    user = User.get(user_id)
    return {
        "id": user.id,
        "name": user.name,
        "email": user.email,
        "preferences": user.preferences,
        "last_login": user.last_login,
        "stats": user.get_stats(),
        # ... 20+ 字段
    }

# ✅ 第一性：按使用场景拆分，只返回该场景必需数据
def get_user_for_display(user_id):
    return User.get(user_id).to_display_dto()  # 仅展示用字段

def get_user_for_edit(user_id):
    return User.get(user_id).to_edit_dto()     # 仅编辑用字段
```

## 与其它原则的关系

| 原则 | 关系 |
|------|------|
| **YAGNI** | 第一性强调「只做当前约束下必要的事」，与 YAGNI 一致 |
| **KISS** | 从基本事实推导时，往往得到更简单的方案 |
| **SOLID / Clean Architecture** | 第一性帮助判断「哪里需要抽象、哪里不需要」，避免为抽象而抽象 |

---

## 检查清单

在设计与实现前可自问：

- [ ] 我是否清楚这个需求/模块要解决的核心问题？
- [ ] 当前方案是来自「基本约束」还是「别人都这么做」？
- [ ] 若去掉某个技术/模式，哪些约束会真正被违反？
- [ ] 是否有更小、可验证的一步能先验证假设？
- [ ] 现有实现里，有没有仅因历史存在、已无实际价值的复杂度？

---

## 小结

第一性原则不是反对使用成熟方案，而是：

- **先厘清问题与约束**，再选技术与架构
- **在满足约束的前提下**，优先简单、可验证的实现
- **持续质疑**「我们一直这样写」和「行业都这样」，让设计服务于当前目标

将其与 SOLID、Clean Code、DRY/KISS/YAGNI 结合使用，能减少过度设计、提高设计与需求的契合度。
